구독 중인 다양한 AI 모델들의 남는 토큰과 리소스를 활용해 '자동화된 AI 개발팀'을 구축하겠다는 아이디어는 리소스 낭비를 막으면서도 생산성을 극대화할 수 있는 매우 훌륭하고 흥미로운 접근입니다.

말씀하신 요구사항을 바탕으로 이 다중 에이전트(Multi-Agent) 시스템을 구체화하기 위한 기술 스택, 개발 방향, 그리고 고려해야 할 개선점들을 정리해 드립니다.

---

### 1. 추천 개발 스택 (Tech Stack)

이 프로젝트는 **GitHub API 통신, 로컬 CLI 프로세스 제어, 복잡한 텍스트 파싱(에이전트 간 대화)** 이 주를 이룹니다. 따라서 AI 생태계가 가장 잘 구축된 **Python** 또는 비동기 처리에 강점이 있는 **Node.js(TypeScript)** 를 추천합니다. 여기서는 AI 에이전트 제어에 압도적으로 유리한 **Python**을 기준으로 제안합니다.

* **주요 언어 & 런타임:** Python 3.10+
* **GitHub 연동:** `PyGithub` (GitHub REST API 제어용)
* **에이전트 오케스트레이션:** `CrewAI` 또는 `Microsoft AutoGen`
* 여러 AI가 역할을 나누고 서로 토론하며 작업을 수행하도록 설계된 프레임워크입니다. 직접 구현할 수도 있지만, 이 도구들을 활용하면 에이전트 간의 멘션(@)과 대화 기록 관리가 훨씬 수월합니다.


* **로컬 CLI 제어:** Python 내장 `subprocess` 모듈 (Copilot-cli, 로컬 스크립트 실행용)
* **Git 버전 관리 제어:** `GitPython` (Python 코드 내에서 Git 명령어 제어) 및 `git worktree` 명령
* **알림 시스템:** `smtplib` (내장 메일 발송) 또는 SendGrid/AWS SES API

### 2. 핵심 개발 방향 및 기술적 제안

#### A. 병렬 처리를 위한 최적의 Git 전략 제안 (`git worktree`)

사용자님께서 제안하신 `git worktree`는 **가장 완벽한 접근 방식**입니다. 에이전트들이 하나의 레포지토리를 clone 받아 동시에 작업하면 로컬 파일 시스템에서 충돌이 발생합니다.

* **방식:** 메인 레포지토리를 한 번만 clone 한 뒤, PM 에이전트가 이슈를 할당할 때마다 `git worktree add ../feature-issue-1 branch-issue-1` 과 같이 독립된 디렉토리(worktree)를 생성합니다.
* **장점:** 각 에이전트(Gemini, Codex 등)는 자신만의 격리된 디렉토리에서 코드를 수정하고 로컬 CLI 툴을 실행할 수 있어 파일 I/O 충돌을 원천 차단합니다. 작업 완료 후 커밋, 푸시하고 PR을 생성한 뒤 해당 worktree 디렉토리를 삭제하면 깔끔하게 관리됩니다.

#### B. 앱 동작 흐름 및 아키텍처 (단계별)

1. **초기화 (Init):** 앱 실행 시 `.env` 파일을 읽어 각 AI CLI의 실행 경로, 이메일, GitHub Token 등을 로드합니다.
2. **모니터링 (Monitor):** 스케줄러(예: `APScheduler`)를 통해 등록된 레포지토리를 주기적으로 폴링(Polling)합니다.
3. **계획 및 설계 (Planning - 최대 3회):**
* `plan.md` 생성 감지 시, GitHub에 "Planning 논의" 이슈를 자동 생성합니다.
* PM(기본 Gemini)이 주도하여 다른 에이전트(Claude, Codex 등)를 멘션(`@claude`)하며 리뷰를 요청합니다.
* GitHub Issue Comment API를 사용해 각 에이전트의 응답을 이슈 댓글로 달아 사용자가 실시간으로 진행 상황을 볼 수 있게 합니다.
* 3회 반복 후 PM이 최종 결과를 요약하여 `final_plan.md`로 업데이트(커밋)합니다.


4. **작업 할당 (Task Allocation):** PM이 `final_plan.md`를 기반으로 하위 Issue들을 생성하고, 에이전트들의 현재 토큰 소모량/부하 상태를 계산하여 댓글로 작업을 배정합니다.
5. **실행 및 리뷰 (Execution & Review - 최대 3회):**
* 각 에이전트는 할당받은 이슈에 대해 `git worktree`로 브랜치를 따서 로컬 CLI를 호출하여 작업을 진행합니다.
* 작업 후 PR(Pull Request)을 생성하고, 코드 리뷰 담당 에이전트를 지정합니다.
* 리뷰에 따른 수정 작업(최대 3회)을 진행한 후, 테스트(로컬 스크립트 실행)를 통과하면 머지(Merge)합니다.


6. **완료 알림 (Done):** 모든 이슈가 닫히면, 환경변수에 등록된 사용자 이메일로 작업 완료 및 요약 리포트를 발송합니다.

### 3. 개선점 및 현실적인 한계 (주의 깊게 설계해야 할 부분)

* **폴링(Polling) vs 웹훅(Webhook):**
* *문제:* "주기적으로 체크한다"는 폴링 방식은 GitHub API 호출 제한(Rate Limit)에 걸리기 쉽습니다.
* *개선안:* 로컬에서 앱을 구동하더라도 `ngrok`이나 Cloudflare Tunnels를 이용해 로컬 포트를 외부로 열고, **GitHub Webhook**을 연동하여 `plan.md` 푸시나 이슈 생성이 발생할 때만 즉각적으로 앱이 동작하도록 하는 것이 훨씬 효율적입니다.


* **토큰 소비량 추적의 어려움:**
* *문제:* API(OpenAI, Anthropic 등)를 직접 호출하면 토큰 사용량을 응답받을 수 있지만, '로컬 CLI 툴(예: copilot-cli)'을 블랙박스 형태로 실행할 경우 정확한 토큰 소모량을 PM이 알아내기 어렵습니다.
* *개선안:* 토큰 추적이 불가한 CLI의 경우 "이슈 1개 처리 = 가중치 N", "API 직접 호출 = 실제 토큰량 기반" 형태로 서로 다른 가중치 시스템을 두어 PM이 할당량을 조율하도록 설계해야 합니다.


* **Merge Conflict (병합 충돌) 처리:**
* *문제:* 여러 에이전트가 병렬로 동일한 파일이나 로직을 수정하다 보면 반드시 Git 병합 충돌이 발생합니다.
* *개선안:* PR 리뷰 과정에서 충돌이 감지되면, PM 에이전트나 코드 리뷰 담당 에이전트가 충돌 파일의 Git 마커(`<<<<<<< HEAD`)를 읽고 해결 방안을 토론하여 수정하는 로직이 반드시 필요합니다.


* **에이전트 무한 루프 방지:**
* 사용자님께서 "3차시까지 논의", "개선 작업 최대 3번"으로 리미트를 설정하신 것은 **매우 훌륭한 설계**입니다. LLM 에이전트들은 종종 쓸데없는 양보("네, 말씀하신 것이 맞습니다")나 칭찬으로 턴을 낭비하므로, 명확한 횟수 제한과 프롬프트 상의 '목적 달성 시 즉시 종료' 지시가 필수적입니다.
